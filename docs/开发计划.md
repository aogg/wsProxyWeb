# WebSocket代理网页请求系统 - AI分步骤开发计划

## 项目概述

本项目实现一个通过浏览器插件和WebSocket代理网页请求的系统：
- **客户端**：浏览器插件（支持Chrome和Edge），拦截网页请求并通过WebSocket发送
- **服务端**：Go语言开发的WebSocket服务，接收请求、执行HTTP请求、返回响应
- **通信**：WebSocket协议，支持数据加密和压缩

## 项目结构说明

**重要**：客户端和服务端完全分离，位于不同的文件夹中：

- **`server/`**：服务端目录（Go项目）
  - 独立的Go模块
  - 使用Go标准项目结构
  - 编译生成可执行文件

- **`client/`**：客户端目录（npm项目）
  - 独立的npm项目
  - 使用webpack/vite等工具打包
  - 支持TypeScript（推荐）
  - 打包后生成浏览器插件

**开发步骤说明**：
- 开发步骤不区分客户端和服务端，按功能模块统一规划
- 但在实现时，需要明确代码属于客户端还是服务端
- 文件路径会明确标注 `server/` 或 `client/` 前缀

## 浏览器兼容性说明

**重要**：客户端插件必须同时支持以下浏览器：
- **Google Chrome** (Manifest V3)
- **Microsoft Edge** (Manifest V3)

Chrome和Edge都基于Chromium内核，使用Manifest V3规范，API基本一致。开发时需要注意：
- 使用标准的Chrome Extension API（Edge完全兼容）
- 避免使用Chrome特有的API（如无必要）
- 测试时需要在两个浏览器中都验证功能

```json
  },
  "action": {
    "default_popup": "popup.html"
  }
}
```

## 第三阶段：请求拦截和转发

### 步骤3.1：实现浏览器插件请求拦截

**任务**：拦截网页的HTTP/HTTPS请求

**文件**：`client/src/background/background.ts` (或.js)

**功能要求**：
1. 使用 `chrome.webRequest.onBeforeRequest` 拦截请求
2. 收集请求信息：URL、方法、Headers、Body
3. 将请求信息转换为协议格式
4. 通过WebSocket发送到服务端

**实现要点**：
- 拦截所有请求或按规则过滤（可配置）
- 处理不同类型的请求（GET、POST等）
- 处理请求Body（可能需要异步读取）

**验收标准**：可以拦截网页请求并发送到服务端

---

### 步骤3.2：实现Go服务端HTTP请求执行

**任务**：接收请求数据，执行真实的HTTP请求

**文件**：`server/src/logic/request_logic.go`, `server/src/libs/http_lib.go`

**功能要求**：
1. 解析客户端发送的请求数据
2. 使用Go的 `net/http` 构造HTTP请求
3. 执行请求并获取响应
4. 将响应数据封装为协议格式
5. 通过WebSocket返回给客户端

**实现要点**：
- 支持常见HTTP方法（GET、POST、PUT、DELETE等）
- 处理请求Headers和Body
- 处理响应Headers和Body
- 错误处理和超时控制

**验收标准**：服务端可以接收请求并返回正确的响应

---

### 步骤3.3：实现浏览器插件响应处理

**任务**：接收服务端响应，还原为原始请求的响应

**文件**：`client/src/background/background.ts` (或.js)

**功能要求**：
1. 接收服务端返回的响应数据
2. 将响应数据还原为浏览器可识别的格式
3. 使用 `chrome.webRequest.onBeforeRequest` 阻止原始请求
4. 返回代理响应

**实现要点**：
- 使用 `chrome.webRequest.onBeforeRequest` 的 `redirectUrl` 或 `cancel` 选项
- 或者使用 `chrome.webRequest.onHeadersReceived` 修改响应
- 处理二进制数据（如图片）

**验收标准**：网页请求可以通过代理正常返回响应

## 开发注意事项

### Chrome和Edge兼容性

1. **API兼容性**：Chrome和Edge都基于Chromium，API基本一致，但需要注意：
   - 使用标准的Chrome Extension API
   - 避免使用实验性API
   - 在两个浏览器中都进行测试

2. **测试建议**：
   - 开发时在Chrome中测试
   - 发布前在Edge中验证
   - 注意版本差异（尽量支持较新版本）

3. **Manifest V3**：
   - Chrome 88+ 和 Edge 88+ 都支持
   - 使用Service Worker替代Background Page
   - 注意Service Worker的生命周期

### 代码组织原则

1. **单一职责**：每个模块只负责一个功能
2. **分层清晰**：
   - 服务端：logic层处理业务逻辑，libs层提供工具功能
   - 客户端：按功能模块组织（background, popup, libs等）
3. **配置分离**：配置文件独立，便于修改
4. **错误处理**：所有关键操作都要有错误处理
5. **独立构建**：客户端和服务端独立构建，互不干扰

### 客户端构建说明

1. **开发环境**：
   - 使用 `npm run dev` 进行开发，支持热重载
   - 修改代码后自动重新打包
   - 在浏览器中重新加载插件即可看到效果

2. **生产构建**：
   - 使用 `npm run build` 进行生产构建
   - 打包后的文件在 `client/dist` 目录
   - 将dist目录加载到Chrome/Edge作为插件

3. **TypeScript支持**（推荐）：
   - 使用TypeScript提供类型检查
   - 编译为JavaScript后打包
   - 提供更好的开发体验

### 开发顺序建议

1. 先实现基础功能（无加密压缩）
2. 验证端到端流程正常
3. 再添加加密压缩
4. 最后完善配置和界面
5. 客户端开发时，每次修改后需要重新打包并重新加载插件

# WebSocket通信协议说明

本文档定义了浏览器插件客户端和Go服务端之间的WebSocket通信协议。

## 协议概述

- **传输方式**：WebSocket（文本消息，JSON格式）
- **编码**：UTF-8
- **消息格式**：JSON对象
- **连接地址**：`ws://localhost:8080/ws`（可配置）

## 基础消息结构

所有消息都遵循以下基础结构：

```json
{
  "id": "消息唯一标识符",
  "type": "消息类型",
  "data": {} // 消息数据（根据类型不同而不同）
}
```

### 字段说明

- **id** (string, 必需)：消息唯一标识符，用于请求和响应的关联。建议使用UUID或时间戳+随机数。
- **type** (string, 必需)：消息类型，见下方消息类型列表。
- **data** (object, 可选)：消息数据，根据消息类型不同而不同。

## 消息类型

### 1. 心跳消息

用于保持连接活跃，客户端定期发送，服务端响应。

#### 客户端 → 服务端：ping

```json
{
  "id": "heartbeat_1234567890",
  "type": "ping",
  "data": {
    "timestamp": 1234567890
  }
}
```

#### 服务端 → 客户端：pong

```json
{
  "id": "heartbeat_1234567890",
  "type": "pong",
  "data": {
    "timestamp": 1234567890
  }
}
```

**说明**：
- 客户端每30秒发送一次ping消息
- 服务端收到ping后立即回复pong
- 如果超过60秒未收到心跳，认为连接断开

---

### 2. HTTP请求代理消息

用于将浏览器的HTTP请求转发到服务端执行。

#### 客户端 → 服务端：request

```json
{
  "id": "req_1234567890_abc123",
  "type": "request",
  "data": {
    "url": "https://example.com/api/data",
    "method": "GET",
    "headers": {
      "User-Agent": "Mozilla/5.0...",
      "Accept": "application/json",
      "Cookie": "session=abc123"
    },
    "body": "", // 对于GET请求为空字符串，POST/PUT等请求包含请求体
    "bodyEncoding": "text" // "text" | "base64" | "binary"，默认为"text"
  }
}
```

**字段说明**：
- **url** (string, 必需)：完整的请求URL
- **method** (string, 必需)：HTTP方法，如 "GET", "POST", "PUT", "DELETE", "PATCH", "HEAD", "OPTIONS"
- **headers** (object, 可选)：HTTP请求头，键值对形式
- **body** (string, 可选)：请求体内容
  - GET/HEAD/OPTIONS请求通常为空字符串
  - POST/PUT/PATCH请求包含请求体
  - 二进制数据使用base64编码
- **bodyEncoding** (string, 可选)：请求体编码方式
  - "text"：文本内容（默认）
  - "base64"：Base64编码的二进制数据
  - "binary"：二进制数据（不推荐，建议使用base64）

#### 服务端 → 客户端：response

```json
{
  "id": "req_1234567890_abc123",
  "type": "response",
  "data": {
    "status": 200,
    "statusText": "OK",
    "headers": {
      "Content-Type": "application/json",
      "Content-Length": "1234"
    },
    "body": "响应内容",
    "bodyEncoding": "text" // "text" | "base64"
  },
  "error": null
}
```

**字段说明**：
- **status** (number, 必需)：HTTP状态码，如 200, 404, 500
- **statusText** (string, 可选)：HTTP状态文本，如 "OK", "Not Found"
- **headers** (object, 可选)：HTTP响应头，键值对形式
- **body** (string, 可选)：响应体内容
  - 文本内容直接返回
  - 二进制数据使用base64编码
- **bodyEncoding** (string, 可选)：响应体编码方式
  - "text"：文本内容（默认）
  - "base64"：Base64编码的二进制数据
- **error** (string | null, 可选)：错误信息，如果请求失败则包含错误描述

**错误响应示例**：

```json
{
  "id": "req_1234567890_abc123",
  "type": "response",
  "data": {
    "status": 0,
    "statusText": "Error",
    "headers": {},
    "body": "",
    "bodyEncoding": "text"
  },
  "error": "网络错误：连接超时"
}
```

### 3. 错误消息

用于通知客户端发生了错误。

#### 服务端 → 客户端：error

```json
{
  "id": "req_1234567890_abc123",
  "type": "error",
  "data": {
    "code": "INVALID_REQUEST",
    "message": "请求格式错误",
    "details": {}
  }
}
```

**字段说明**：
- **code** (string, 必需)：错误代码
  - "INVALID_REQUEST"：请求格式错误
  - "NETWORK_ERROR"：网络错误
  - "TIMEOUT"：请求超时
  - "UNKNOWN_ERROR"：未知错误
- **message** (string, 必需)：错误描述
- **details** (object, 可选)：错误详情

---

### 4. 连接控制消息

用于控制连接状态。

#### 客户端 → 服务端：close

```json
{
  "id": "close_1234567890",
  "type": "close",
  "data": {
    "reason": "用户主动断开"
  }
}
```

#### 服务端 → 客户端：close

```json
{
  "id": "close_1234567890",
  "type": "close",
  "data": {
    "reason": "服务器关闭"
  }
}
```

## 协议实现要求

### 客户端实现要求

1. **消息ID生成**：确保每个请求都有唯一的ID，建议使用 `req_${timestamp}_${random}` 格式
2. **请求关联**：保存请求ID和原始请求的映射关系，以便将响应关联到原始请求
3. **超时处理**：如果30秒内未收到响应，认为请求超时
4. **错误处理**：正确处理服务端返回的错误消息
5. **心跳机制**：每30秒发送一次ping消息

### 服务端实现要求

1. **消息解析**：正确解析客户端发送的JSON消息
2. **请求执行**：根据请求数据构造HTTP请求并执行
3. **响应封装**：将HTTP响应封装为协议格式返回
4. **错误处理**：捕获所有错误并返回错误消息
5. **心跳响应**：收到ping消息后立即回复pong
6. **超时控制**：HTTP请求超时时间建议设置为30秒

## 注意事项

1. **消息顺序**：WebSocket保证消息顺序，但多个请求可能并发执行，响应顺序可能与请求顺序不同
2. **二进制数据**：所有二进制数据必须使用Base64编码传输
3. **字符编码**：所有文本数据使用UTF-8编码
4. **大小限制**：建议单个消息不超过10MB，超过时考虑分块传输（后续版本实现）
5. **安全性**：生产环境应启用加密和压缩（第四阶段实现）
